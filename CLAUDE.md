## What This Project Is

Glitch API is a **local-first external brain / knowledge management system**. It ingests documents from the filesystem, extracts text via OCR, generates vector embeddings for semantic search, and models entities (tasks, journals, goals, etc.) as nodes in a knowledge graph. Two processes run concurrently:

1. **Frontend + API server** (TanStack Start, port 3000) — SSR React app with Drizzle ORM for database access
2. **Workflow engine** (Motia, port 3001) — event-driven pipeline for document ingestion, OCR, and embedding generation

## Commands

```bash
# Initial setup
npm install
npm run db:up        # Start PostgreSQL + Redis in Docker
npm run db:migrate   # Apply migrations

# Development (two terminals)
npm run dev          # TanStack Start frontend/server (port 3000)
npm run workflow:dev # Motia workflow engine (port 3001)

# Database
npm run db:generate  # Generate migration files after schema changes
npm run db:migrate   # Apply migrations
npm run db:studio    # Open Drizzle Studio (visual DB browser)
npm run db:up / db:down  # Start/stop Docker containers

# Build
npm run build
npm run start

# Workflow build
npm run workflow:build
```

No test runner is configured yet.

## Architecture

### Data Flow
```
Filesystem → document-watcher (Motia Service)
           → emits document:detected
           → ocr-processor (Motia Step)
           → emits ocr:completed
           → embedding-generator (Motia Step)
           → stores chunks + vectors in DB
```

### Key Directories

- **`db/schema.ts`** — Single source of truth for all Drizzle table definitions and TypeScript types. All schema changes start here.
- **`db/migrations/`** — SQL migrations generated by `db:generate`. The initial migration sets up pgvector, all tables, indexes, and `updated_at` triggers.
- **`workflows/`** — Motia workflow steps. Each file exports a step that subscribes to and/or emits typed events. The `document-watcher.ts` is a long-running service (not a one-shot step).
- **`app/routes/`** — TanStack Router file-based routes. `__root.tsx` is the layout; `$id.tsx` patterns are dynamic segments.
- **`scripts/migrate.ts`** — Standalone migration runner (invoked by `db:migrate`).

### Database Design Patterns

- All tables use UUID primary keys (`crypto.randomUUID()` as default)
- `entities` is the central polymorphic table — `type` enum differentiates TASK, JOURNAL, LOG, REMINDER, KB_ARTICLE, GOAL, FINANCIAL_DOC
- `embeddings` stores 384-dimensional vectors (`vector(384)`) with an HNSW index using cosine distance — matches the `all-MiniLM-L6-v2` model dimension
- `links` creates a knowledge graph between entities — enforces no self-links and unique (source, target, relationship) triples
- `documents` tracks ingested files with MD5 hash for deduplication and `ocr_status` enum for pipeline state

### Path Aliases (tsconfig.json)

```
@/*        → root
@/app/*    → app/
@/db/*     → db/
@/workflows/* → workflows/
```

### Environment Variables

See `.env.example` for the full list. Key ones:

```
DATABASE_URL          # Full Postgres connection string
WATCH_PATHS           # Comma-separated dirs for document-watcher
EMBEDDING_MODEL       # Default: all-MiniLM-L6-v2
EMBEDDING_DIMENSION   # Default: 384 (must match vector(384) in schema)
OCR_ENGINE            # Default: paddleocr
```
